# Session Runs & Execution Data

This directory serves as the volatile storage and workspace for all session-specific execution data generated by the Counter-Strike coach AI. It acts as a temporary buffer for active processing tasks and a historical record for completed analysis cycles.

## Technical Overview

The `runs/` directory is designed to handle high-volume data throughput during match ingestion and model training. Each execution cycle (a "run") creates a timestamped or ID-based subdirectory to isolate its data from other sessions. This isolation ensures that intermediate analysis results and training checkpoints do not overwrite each other, allowing for concurrent processing of multiple demos or training sessions.

## Key Components

- **Training Checkpoints**: During model fine-tuning or reinforcement learning cycles, periodic state saves (weights, optimizer states) are stored here.
- **Intermediate Analysis Results**: Temporary JSON and binary files generated during the parsing of demo files before they are aggregated into the final database or report.
- **Raw Session Logs**: Detailed, low-level execution logs specific to a single run, useful for debugging failed ingestions or model drift.
- **Inference Cache**: Transient data used during VLM/LLM inference to speed up repetitive queries within the same session.

## Directory Structure

```text
Programma_CS2_RENAN/runs/
├── [run_id_or_timestamp]/  # Isolated workspace for a specific session
│   ├── checkpoints/         # Model weights and training state
│   ├── intermediate/        # Partially processed demo data
│   └── session.log          # Detailed log for this specific run
├── README.md                # This documentation
├── README_IT.md             # Italian version
└── README_PT.md             # Portuguese version
```

## Usage

1. **Active Processing**: When a new demo ingestion starts, the system automatically creates a new folder within `runs/` to store temporary state.
2. **Model Training**: The training script outputs its progress and periodic `.pth` or `.ckpt` files into this directory.
3. **Cleanup**: Because this storage is considered volatile, it is recommended to archive important checkpoints and clear old run folders periodically to save disk space. The system includes automated cleanup policies for runs older than a specific threshold.
4. **Debugging**: In the event of an engine crash, the files within the specific run folder are the primary source for post-mortem analysis.
