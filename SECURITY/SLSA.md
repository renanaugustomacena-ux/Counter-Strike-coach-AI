## 0. What is SLSA?

[SLSA](https://slsa.dev) (Supply-chain Levels for Software Artifacts) v1.0 defines a graduated set of
requirements for build-system trustworthiness. The Build Track is what matters for us today: it
constrains how an artifact (`.exe`, `.tar.gz`) is produced and what evidence accompanies it.

| Track | What it claims |
|---|---|
| **Build L1** | Provenance is generated and distributed |
| **Build L2** | Provenance is signed; tamper-resistant against the build system itself in transit |
| **Build L3** | Provenance is generated by a hosted, isolated build platform; cryptographically resistant against the **builder** being compromised |
| **Build L4** | Reproducible / hermetic builds; auditable from source to artifact byte-for-byte |

We target **Build L3** for v1.0 release. L4 is evaluated in Phase 4.

## 1. Build-Platform Threat Model

### 1.1 Threats

| ID | Threat | Mitigated by SLSA L3? |
|---|---|---|
| BP-1 | Source code tampered between commit and build | Yes — provenance binds source ref to artifact |
| BP-2 | Dependency compromised post-resolution (lockfile mutation) | Yes — `--require-hashes` + provenance |
| BP-3 | Build runs an unauthorised script that injects code | Yes — runner is hosted GitHub Actions; sandboxed; provenance records exact workflow steps |
| BP-4 | Builder compromise (rare) | Yes — Sigstore/Rekor attestation is verifiable by users without trusting the builder |
| BP-5 | Artifact mutated post-build before upload | Yes — provenance hash binds artifact to build event |
| BP-6 | Release signed with leaked code-signing key | **Out of SLSA scope** — covered by EV-cert rotation (C-SC-08) |
| BP-7 | Reproducibility — independent rebuild from same source byte-equal? | **L4 only** — partial today (PyInstaller embeds timestamps) |

### 1.2 Trusted parties

- **GitHub Actions hosted runners** — trusted to run workflow steps in isolation
- **Sigstore Rekor transparency log** — trusted as immutable record
- **Repository owner's signing identity** — trusted
- **PyPI / GHCR / HuggingFace** — partially trusted, mitigated via hash-pinning + revision-pinning + digest-pinning

## 2. Implementation

### 2.1 Workflow

A new `.github/workflows/release.yml` (decoupled from `build.yml`):

```yaml
name: release
on:
  push:
    tags: [ "v*.*.*" ]
permissions:
  contents: write       # to upload artifacts
  id-token: write       # to use OIDC for Sigstore
  attestations: write   # to publish attestations
jobs:
  build:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@<sha>
      - uses: actions/setup-python@<sha>
      - run: pip install --require-hashes -r requirements-lock-cpu.txt
      - run: python -m PyInstaller --noconfirm packaging/cs2_analyzer_win.spec
      - run: python tools/audit_binaries.py
      - id: hash
        run: echo "digest=$(python -c 'import hashlib,sys; print(hashlib.sha256(open(sys.argv[1],"rb").read()).hexdigest())' dist/Macena_CS2_Analyzer/Macena_CS2_Analyzer.exe)" >> "$GITHUB_OUTPUT"
      - uses: actions/attest-build-provenance@<sha>
        with:
          subject-path: dist/Macena_CS2_Analyzer/Macena_CS2_Analyzer.exe
      - run: cyclonedx-py environment -o sbom.cdx.json
      - uses: actions/attest-sbom@<sha>
        with:
          subject-path: dist/Macena_CS2_Analyzer/Macena_CS2_Analyzer.exe
          sbom-path: sbom.cdx.json
      # Code signing (when EV cert is procured — D-1)
      - run: signtool sign /v /tr http://timestamp.digicert.com /td sha256 /fd sha256 /a "dist\Macena_CS2_Analyzer\Macena_CS2_Analyzer.exe"
        if: env.EV_CERT_AVAILABLE == 'true'
      - uses: actions/upload-artifact@<sha>
        with:
          name: cs2-analyzer-windows
          path: dist/Macena_CS2_Analyzer/
```

### 2.2 End-user verification

Users can verify a downloaded artifact:

```bash
gh attestation verify cs2-analyzer.exe --repo <owner>/<repo>
```

This:
- Fetches the attestation bundle from GitHub
- Verifies the Sigstore signature against the Rekor transparency log
- Confirms the artifact hash matches the attested hash
- Confirms the build was triggered by a tag push on the repo

### 2.3 SBOM verification

```bash
gh attestation verify cs2-analyzer.exe --repo <owner>/<repo> --predicate-type cyclonedx
```

## 3. Gap Analysis vs SLSA Build L4

L4 requires:
- **Hermetic builds** — no network access during build, all inputs declared upfront
- **Reproducible** — independent build of the same source produces byte-identical output

**Current gaps:**
- PyInstaller embeds Windows file-system timestamps and ASLR random-base hints in the `.exe` binary,
  preventing byte-identical reproduction.
- Hosted GitHub Actions runners have network access during `pip install` (mitigated by `--require-hashes`,
  but not network-disabled).

**Phase 4 evaluation path:**
- **Option A** Nix flake (`flake.nix`) for hermetic build — drops GH-Actions hosted runner in favour
  of Nix-managed dependencies. High effort.
- **Option B** Bazel with hermetic Python toolchain (`rules_python` + manifest of pre-fetched wheels).
  Medium effort.
- **Option C** Accept the gap; document SLSA L3+ posture; stay at "L3 with hash-pinned reproducibility
  best-effort."

Recommendation (deferred to Phase 4): start with `tools/verify_reproducibility.py` to measure the
actual reproducibility delta (re-run two builds on different runners, diff `.exe`). The diff size
informs whether L4 is worth the migration cost.

## 4. Phase Plan

### Phase 1 (now — non-invasive)
- This document.
- `SECURITY/SLSA.md` describing target posture.

### Phase 2 (after live ingestion completes)
- `--require-hashes` enabled in CI (C-SC-03) — prerequisite for trustworthy provenance.

### Phase 3 (release readiness)
- `release.yml` with `actions/attest-build-provenance@<sha>` and `actions/attest-sbom@<sha>`.
- First v1.0.0 release tag → Sigstore-signed provenance + SBOM attestation.

### Phase 4 (long horizon)
- Reproducibility measurement.
- Evaluate Nix / Bazel for L4.

## 5. Decision Log (per Doctrine §67)

| Decision | Risk addressed | Residual | Tradeoffs | Assumptions |
|---|---|---|---|---|
| Target SLSA Build L3, not L4 | BP-1…BP-5 contained | BP-7 reproducibility — best-effort | Cost of migration to Nix is multi-week | PyInstaller remains the packager |
| Use GitHub Actions native attestation actions over manual `cosign` | Vendor-supported, audited | Tied to GitHub's OIDC + Sigstore | None significant for this scale | Project remains on GitHub |
| Decouple `release.yml` from `build.yml` | Confines secrets to release job; minimises blast radius | More YAML to maintain | Two pipelines | Release frequency is moderate |
| Code signing in same job as attestation | Both produced with one OIDC token | If signing fails, release is unsigned | EV cert not yet procured | Budget approval pending (D-1 in plan) |
